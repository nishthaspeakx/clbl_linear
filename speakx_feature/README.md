# SpeakX English Town — Vertical Slice (Progress)

Reference scaffold proving the migration architecture end-to-end on the SpeakX
stack: **Flutter (Bloc) → gRPC → Golang → MongoDB**. Built from the RN prototype
in `../src`. See `../docs/FLUTTER_MIGRATION_PLAN.md` for the full plan.

This slice covers ONE feature — **Progress** — through every layer, so the team
can validate conventions and estimate the rest before committing to all 147 files.

```
speakx_feature/
├── proto/english_town.proto            # gRPC contract (Progress today)
├── backend/englishtown/                # Golang service
│   ├── store.go                        # Mongo repository (user_progress)
│   └── service.go                      # gRPC handlers + proto mapping
└── flutter/lib/features/english_town/  # Flutter feature (clean architecture)
    ├── domain/      entities + repository interface
    ├── data/        datasources (gRPC remote + prefs cache) + repo impl
    └── presentation/ cubit + progress_screen
```

## Data model (1:1 with the prototype)
`Progress { currentId: int, completedIds: int[], coins: int }`
— matches `src/storage/progressStorage.ts`, whose own comments mark this exact
API seam.

## To make it run (team checklist)
1. **Generate stubs from the proto**
   - Go: `protoc --go_out=. --go-grpc_out=. proto/english_town.proto`
   - Dart: `protoc --dart_out=grpc:flutter/lib/generated proto/english_town.proto`
2. **Backend:** mount `englishtown.NewService(store)` on SpeakX's gRPC server;
   pass the real `*mongo.Database`; wire the real auth interceptor
   (replace `userIDFromContext`).
3. **Flutter:** uncomment the generated-client calls in
   `progress_remote_datasource.dart`; register `ProgressRepositoryImpl` in
   SpeakX's DI (get_it/injectable) so `ctx.read<ProgressRepository>()` resolves.
4. **pubspec.yaml** deps: `flutter_bloc`, `bloc`, `equatable`, `grpc`,
   `protobuf`, `shared_preferences`.

## ⚠️ Assumptions to confirm
- State mgmt = **Bloc/Cubit**, transport = **gRPC** (per your answers).
- Routing, theming, DI, and auth interceptor follow SpeakX's existing patterns —
  swap the placeholders for the real ones.

## Why these are stubs, not runnable yet
The gRPC client/server types are generated by `protoc` (step 1) and the repo
must plug into SpeakX's DI + auth, which live in the app I don't have access to.
Everything around those two seams is real, idiomatic code ready to wire.
